Abstract

Model-Based User Interface Design facilitates interchange of designs through a layered approach that separates out different levels of abstraction in user interface design. This document covers the specification of Abstract User Interface Models, by defining its semantics through a meta-model, and an interchange syntax (expressed as XML Schema) for exchanging Abstract User Interface Models between different user interface development environments.

Introduction

This document aims at defining a standard meta-model for expressing an Abstract User Interface (AUI) as specified in the Cameleon Reference Framework (CRF). Its specific goals include: reviewing the state-of-the-art in the domain of interest, defining the main concepts for the meta-model, as well as their properties and relationships, identifying its main requirements, proposing, describing and discussing a meta-model that provides a standard definition for AUI models. Such proposal will be validated by its instantiation according to the use case scenarios presented in the MBUI Introduction document. By defining a standard meta-model for AUIs, several potential benefits are expected: ensuring some compatibility and interoperability among formats for expressing AUIs, defining a unified vocabulary, and a consistent methodological approach for specifying an AUI.

The meta-model of this document represents both an abstraction and a generalisation of the meta-models existing in the literature. As an abstraction, it provides an abstract syntax based on a rationalised selection of main concepts, to which existing or futureUIDLs that are compliant tothis meta-model, can be mapped. This ensures that AUIs specified with different UIDLs could be related to each other, thus enabling interoperability. As a generalisation, the meta-model of this documentprovides an expressive set of constructs thatcan be directly exploited to develop tools by implementing and extending the classes of thismeta-model. Again, specification fragments developed with such tools would support interoperability with other fragments coming from compliant tools. A UIDL or a software for specifying an AUI is said to be AUI compliantif and only ifit translates, imports, or exports, without residuals, an AUI specified according to the XML Schema derived from the meta-model.

Definition

The Abstract User Interface (AUI) (corresponding to the Platform-Independent Model –PIM– in Model-Driven Engineering) is an expression of a UI in terms of interaction units without making any reference to implementation neitherin terms of interaction modalities nor in terms of technological space (e.g., computing platform, programming or markup language). Therefore, an AUI is said to be independent of any interaction modality (e.g., graphical, vocal, haptic, 3D) and independent of which interactors will be available for its implementation in any particular programming or markup language. An Abstract User Interface is composed of Abstract Interaction Units (AIUs) that supporta set of (sub-)tasks that are logicallyrelated from the user’s perspective in order to achieve a particular user’s goal. Different synonyms have been used over history to refer to an abstract interaction unit: presentation unit, interaction space, workspace.

An AUI maybe connected upwards to one or severaltask modelsand/or downwards to one or severalConcrete User Interface Models. This connection canbe achieved via different mechanisms, such as mapping or transformation either at design-time or at run-time. An AUI could also exist in isolation, independently of theexistence of any other related model. In that case, an AUI represents a possible entry-point for further design and development of the UI of an interactive application and gives rise to one or many Final User Interfaces.

Specific Requirements

This section revisits the common generic requirements defined for all meta-models by refining their definition in the scope of the AUI meta-model:

  • GR1: Completeness. An AUI model must be expressive enough to define, describe, and analyse any user interface without making any reference to a particular interaction modality or implementation. By definition, an AUI is independent of any context of use, but may be refined in a way that would re-organize its contents depending on constraints imposed on the final user interfaces. An AUI definition should be self-contained enough in order to specify its contents and should not make any reference to any other CRF level, such as task, Concrete User Interface (CUI) or Final User Interface (FUI). However, a same task model may give rise to one or many AUIs, one AUI may give rise to one or many Concrete User Interface (CUI) models or to one or more Final User Interfaces subsequently. Addressing this requirement may subsume the following sub-requirements
    • SR1.1: AUI structure. an AUI should be composed of abstract interaction units at different levels of decomposition.
    • SR1.2: AUI behaviour. an AUI should be attached to the expression of one or more behaviours.
  • GR2: Consistency. Regarding the AUI, consistency means the ability to agree and follow the definitions that were previously specified.
  • GR3: Separation of concerns. Regarding the AUI, separation of concerns means the ability to separately define the various AUI concerns (e.g., structure, presentation, behaviour) in an isolated manner without overlapping.For any concept already defined, the AUI should avoid redefining it. Addressing this requirement may subsume the following sub-requirements:
    • SR3.1: Separation of static vs dynamic aspects. the AUI static definition (i.e. its structure) should be clearly separated from the dynamic definition (i.e. its behaviour).
    • SR3.2: Autonomy of static vs dynamic aspects. static and dynamic aspects should be specified independently of each other in order to enable autonomy in their respective specification. For instance, an AUI could contain a static definition without requiring a dynamic definition ; for a same static definition, different dynamic definitions could be specified independently of each other.
  • GR4: Extensibility. Any AUI could to be easily extended with new elements that describe new concepts.
  • GR5: Correlability. Regarding the AUI, correlability means that a AUI should be able to belinked to related models (e.g., domain model for data binding, context model if needed to express the applicability of an AUI for a specific context of use) and mapped to other levels (e.g., upwards to a task model, downwards to a Concrete User Interface, or at the same level of abstraction to another related AUI, for example in a different, but related, context of use.
  • GR6: Concision. Regarding the AUI, concision means that the meta-model should be as concise andcompact as possible. This could be achieved by different means such as minimizing the amount of classes, reducing the size of the meta-model, factoring out common properties when possible.

UserInterfaceModel (UICommons)

Definition: defines a model for a UserInterfaceModel

Attributes:

  • none

Relationships:

  • aggregates(AbstractUserInterfaceModel): a UserInterfaceModel is defined by an AbstractUserInterfaceModel
  • aggregates (UserInterfaceModelElement): a UserInterfaceModel is defined by an UserInterfaceModelElement

AbstractUserInterfaceModel

Definition: this model describes an Abstract User Interface (AUI) in terms of its structure (see AbstractInteractionUnit) and its model (see UserInterfaceModel).

Attributes:

  • none

Relationships:

  • inheritsFrom (UICommons: UserInterfaceModel): since an AUI model is a particular type of Model, it therefore inherits from all its attributes and methods.
  • isAggregatedIn (UICommons: UserInterfaceModel): a AUI model is one possible model contained in a general UI model. The AUI model is optional in a UI model: some development path may not require the existence of an AUI model in order to produce a FUI.
  • aggregates (AbstractInteractionUnit): a AUI model consists of a non-empty hierarchy of Abstract Interaction Units (AIUs) in order to define its structure. An AUI model should consist of at least oneAbstract Interaction Unit

AbstractInteractionUnit

Definition: consists of the basic unit for expressing any piece of interaction in a recursive decompositionof an AUI in abstract terms. An Abstract Interaction Unit is typically configured in terms of Abstract Interaction Unit (AIU)based on a task model. This decomposition could be context-independent or linked to one to many context-dependent definitions. Since any level of the recursive decomposition is considered as an Abstract Interaction Unit, the same term is used both for elementary andcompound ones.

Attributes:

  • id (string, 1-1): specifies a unique identifier of the AIU
  • name (string, 1-1): specifies a logical name of the AIU that should not necessarily be unique. For instance, “PersonIdentificationAIU” in order to provide any way to identify a person in a database.
  • role (string, 0-n): specifies the role played by the AIU. For instance, grouping, guidance, error management, navigation, repetition, confirmation.
  • state (string, 0-1): specifies the AIU state.
  • frequency (float, 0-1): specifies the frequency of usage per time unit. This unit is user-defined. For instance, frequency=”5/hour” means that this AIU shouldbe performed five times per hour.
  • repetitionFactor (integer, 0-1): specifies how many times a particular AIU is repeated in a composition with a parent AIU. For instance, repetitionFactor=”2” means that a customer identification could be repeated twice.
  • domainReference (string, 0-1): specifies a data binding with a domain model containing classes, attributes to be manipulated in the AIU

Methods:

  • setCompound (compound: boolean): specifies a composition given a specific compound
  • isCompound (boolean, 1): specifies whether the AIU is compound, therefore being involved in a Composition relationship.

Relationships:

  • isAggregatedIn (AbstractUserInterfaceModel): an Abstract Interaction Unit is part of one and only one Abstract User Interface model.
  • Inheritsfrom (UserInterfaceModelElement): generalizes the UserInterfaceModelElement
  • constrains (): an AbstractInteractionUnit can be restricted according to zero or many constraints defined in the Behaviour package.
  • isComposedby (PresentationSupport): one or several PresentationSupport (interface) can compose the AbstractInteractionUnit
  • isComposedby (EventSupport): one or several EventSupport (interface) can compose the AbstractInteractionUnit
  • isAggregatedin (Composition): an Abstract Interaction Unit is part of one and only one Composition.
  • isAssociatedwith (Composition): an AIU can be associated in a recursive decomposition

DataInputOutputSupport

Definition: consists of an elementary AIU that is responsible for data input and/or output. This AIU could be linked to a domain via a domain reference in order to ensure data binding or could be left out to interact with data without necessarily giving the associated domain model.

Attributes:

  • minCardinality (integer, 0-1): specifies the minimum number of items in the AbstractDataIU.
  • maxCardinality (Integer, 0-1): maximum number of items in the AbstractDataIU.
  • defaultValue (string, 0-n): possible default value or concatenation of default values that can be used if the required value is not available in the Domain Model.
  • dataType (enumDataType): defines the data type for required for the data manipulated in the AIU
  • dataLength (integer, 0-1): defines the maximum length (size) of the data
  • dataFormat (string, 0-1): defines a restriction for a specific format for the data through a mask expressed as a regular expression.
  • Range (string): defines an interval for the data
  • IOFacet (character):
  • isDynamic (boolean): defines if the data is static or dynamic (in terms of changes along time)
  • isRelevant (boolean): defines the existing relevancy level of the data
  • patternMatching (string): defines a restriction for the data (e.g. through a regex)

Relationships:

  • implements (PresentationSupport): is abstracted as PresentationSupport
  • implements (EventSupport): is abstracted as EventSupport
  • instantiates (InputEvent): depends on and uses an InputEvent

DataSelectionSupport

Definition: represents a way to interact with the interface by selecting zero, one or many items among the options available in a list.

Attributes:

  • start (string): starting index (to define the range for selection)
  • end (string): ending index (to define the range for selection)
  • step (string): interval between two successive items, by starting by start and ending by end (for defining the selection options)
  • selectionType (string): defines the characteristic of the selection, e.g. single, multiple and the constraints imposed if any
  • isExpandable (Boolean):specifies if the user can add item in the selection
  • isContinuous (Boolean): specifies if the selection is continuous
  • isRating (Boolean): defines whether a score is associated to the selection (for instance, a score rating a service, an information)
  • isSecret (Boolean): to define whether the values should be hidden

Relationships:

  • implements (PresentationSupport): is abstracted as PresentationSupport
  • implements (EventSupport): is abstracted as EventSupport
  • specializedIn (DataFilterSupport): is generalized by DataFilterSupport
  • instantiates (SelectionEvent): uses and depends of a SelectionEvent
  • instantiates (DeselectionEvent): uses and depends of a DeselectionEvent

DataFilterSupport

Definition: provides support for filtering a data to be selected. A filter is defined as a higher-order function that processes a data structure (typically a list) in order to produce a new data structure containing exactly those elements of the original data structure for which a given filter condition is satisfied.

Attributes:

  • filterType (char): defines the type of filter
  • filterCondition (constraint): defines a filter condition that needs to be satisfied when the filter is applied
  • filterProperty (string): defines properties for the filter

Relationships:

  • inheritsFrom (DataSelectionSupport): is generalized by a DataSelectionSupport

TriggerSupport

Definition: is an Interaction unit that allows to trigger an event.

Attributes:

  • triggerExpression (string): defines an expression for specifying the triggering
  • triggerType (string): defines the type of the triggering support
  • requiresConfirmation (Boolean): defines whether a confirmation is required or not

Relationships:

  • implements (EventSupport): is abstracted by an interface of EventSupport
  • instantiate (TriggerEvent): uses and depends on TriggerEvent
